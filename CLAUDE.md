# VaultForge Claude 总控协议

本文件是 Obsidian Vault 在 Claude、Claude Code 或 Claudian 环境中的根级操作协议。

它负责控制任务路由、权限边界、上下文读取、质量门槛、长期记忆更新和停止条件。项目专属信息放在 `00_System/`，岗位行为放在 `09_SKILLS/`。

## 1. 核心原则

```text
先判断任务，再读取资料。
先确认授权，再执行操作。
只读取当前任务需要的上下文。
先通过质量审查，再更新长期记忆。
完成当前任务后停止。
```

默认行为：

- 分析、审阅、诊断、检查、审计类任务默认只读。
- 写作、修改、更新、整理类任务只处理用户明确授权的范围。
- 未明确授权批量任务时，只处理一个最小可用任务单元。
- 不自动进入下一章、下一篇、下一项、下一阶段。
- 不删除、不重写、不重排无关内容。
- 发现事实冲突时，保留冲突并请求确认，不自行覆盖。

## 3. 任务路由

执行任务前，先判断任务类型，并只读取对应岗位文件。

| 任务 | 岗位文件 |
|---|---|
| 规划、大纲、结构设计 | `09_SKILLS/CreateOutline/SKILL.md` |
| 正文、文章、报告、内容生成 | `09_SKILLS/WriteChapter/SKILL.md` |
| 审稿、评分、质量诊断 | `09_SKILLS/ReviewChapter/SKILL.md` |
| 更新项目记忆、索引、状态 | `09_SKILLS/UpdateProject/SKILL.md` |
| 全局审计、阶段复盘 | `09_SKILLS/AuditProject/SKILL.md` |

一个阶段只使用一个岗位身份。

不得在写作阶段做正式审稿；不得在审稿阶段擅自改正文；不得在项目更新阶段扩写内容。

## 4. 授权边界

### 4.1 默认单任务

用户未明确授权批量时，只处理当前指定对象。

以下情况不构成下一任务授权：

- 后续计划已经存在。
- 项目本身目标很长。
- 当前任务自然引出了下一步。
- 用户只说“继续”，但未说明新范围。
- 之前曾说过“推进项目”。

“继续”默认表示继续完成当前任务，不表示进入下一项。

### 4.2 批量任务

批量任务必须同时满足：

1. 用户明确要求批量处理。
2. 有清晰起止范围。
3. 明确允许连续进入下一项。
4. 单批规模合理。

任一任务失败、阻塞或质量未通过，立即停止并汇报。

### 4.3 修改权限

只修改当前任务直接需要的文件。

不得：

- 未授权重写旧稿或旧记录。
- 未授权删除内容。
- 把未来计划登记为当前事实。
- 把低置信推断升级为正式设定。
- 顺手修无关问题。
- 为掩盖上下文缺失而脑补身份、动机、关系、规则或世界观。

## 5. 上下文读取

### 5.1 单一事实源

优先读取：

- 项目配置：`00_System/ProjectConfig.md`
- 当前状态：`00_System/ProjectState.md`
- 写作规则：`00_System/WritingRules.md`
- 语言规则：`00_System/LanguageRules.md`
- 平台规则：`00_System/PlatformRules.md`
- 索引表：`00_System/CharacterRegistry.md`、`00_System/EntityRegistry.md`
- 当前岗位：`09_SKILLS/.../SKILL.md`

规则冲突时，以更具体、更严格、更新日期更近者为准。

### 5.2 最小上下文切片

单任务默认只读取：

- 当前任务说明。
- 当前对象的直接上下文。
- 最近相关产出或更新记录。
- 相关角色、实体、地点、组织、术语和规则。
- 与当前任务有关的主线、支线、伏笔、矛盾或风险。

默认不得全文读取：

- 全部历史产出。
- 全部角色文件。
- 全部审稿记录。
- 全部世界观设定。
- 已闭环且无关的支线和伏笔。

只有在信息缺失、存在冲突或用户要求全局审计时，才扩大读取范围。

### 5.3 阶段复用

同一任务闭环中：

- 审稿复用写作阶段已加载的上下文。
- 返修只读取审核问题和受影响内容。
- 项目更新复用产出、审稿和相关状态。
- 文件未变化时不得重复全文读取。

## 6. 内容生产闭环

用户要求完整执行时，默认流程：

```text
准备上下文
→ 生成产出
→ 检查硬指标
→ 质量审查
→ 必要时返修
→ 更新长期项目记忆
→ 汇报并停止
```

用户只要求其中一步时，只执行该步骤。

## 7. 阻塞条件

遇到以下情况必须停止：

- 必要输入缺失。
- 已有事实冲突。
- 任务需要新增核心设定、重大规则或改变已有解释。
- 范围模糊，错误假设会带来风险。
- 两轮返修后仍未通过质量门槛。

阻塞格式：

```text
状态：阻塞
任务：
原因：
缺失或冲突：
已检查：
需要用户确认：
建议下一步：
```

## 8. 新增信息生命周期

所有新增角色、实体、地点、组织、物品、能力、规则、术语、事件和伏笔，都遵循：

```text
发现 → 捕获 → 分级 → 登记 → 观察 → 升级 / 休眠 / 废弃
```

推荐状态：

| 状态 | 含义 |
|---|---|
| 临时出现 | 当前产出短暂使用 |
| 观察中 | 可能复用，暂不正式化 |
| 待升级 | 具备功能，等待确认 |
| 正式设定 | 长期项目事实 |
| 休眠 | 暂时不用但保留 |
| 退场 | 已完成功能 |
| 废弃 | 不再使用，仅保留历史 |

以下事项必须用户确认：

- 正式角色、正式组织、核心地点、重要术语。
- 新世界观规则或系统规则。
- 改变已有解释。
- 重大伏笔回收。
- 删除或废弃长期资产。
- 改变项目主方向的决策。

## 9. Registry 规则

Registry 是长期索引。

用途：

- 写作前避免重复创造同功能实体。
- 审稿时检查连续性。
- 更新时只改当前任务实际影响的条目。
- 发现冲突时记录冲突，不自行覆盖。

## 10. 质量审查

审查必须有证据、具体、保守。

推荐 Gate：

1. 完整性：必需结构、元数据、正文、格式是否存在。
2. 任务履约：是否符合用户要求和规划。
3. 使用体验：是否清晰、可读、可执行。
4. 连续性：事实、术语、人物、时间、关系是否一致。
5. 风格：是否符合项目规则，是否有明显 AI 腔。
6. 新增信息：是否捕获、分级，是否越权升级。

不得只写“加强节奏”“优化表达”这类空话，必须给证据和修改目标。

## 11. 项目记忆更新

只有产出通过审查后，才更新长期记忆。

规则：

- 记录已经发生的事实，不记录未来计划。
- 推断必须标记低置信或待确认。
- 冲突进入矛盾或问题记录。
- 先捕获新增信息，再决定是否升级。
- 不编辑无关索引、规则或归档。

## 12. 审计

长项目应定期审计。

检查：

- 项目状态漂移。
- Registry 漂移。
- 角色或实体断线。
- 未解决矛盾。
- 超期伏笔或未兑现承诺。
- 活跃支线过多。
- 待确认事项长期悬挂。
- 质量波动。
- 重复设定或世界观膨胀。

审计默认只读，除非用户明确授权修复。

## 13. 写作支持包

如果存在 `10_WritingSupport/`，它只是辅助层。

优先级：

1. `00_System/` 项目规则。
2. 当前规划、角色、主线和伏笔。
3. `10_WritingSupport/` 中的抽象方法。
4. 模型临场发挥。

只使用可迁移方法，不复制外部作品的专名、关系、连续情节或原文句式。

## 14. 完成汇报

完成任务时汇报：

```text
状态：
任务：
处理范围：
修改文件：
质量结果：
待确认事项：
建议下一步：
```

只读任务的 `修改文件` 写“无”。

## 15. 最终准则

```text
窄读取。
慎修改。
先捕获，再升级。
先审查，再更新。
先停止，再继续。
```
