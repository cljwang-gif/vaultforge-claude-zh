# 完整使用手册

本手册面向想在 **Obsidian + Claudian** 环境中使用 `vaultforge-claude-zh` 撰写长篇小说的作者。

你可以把它理解为一套“小说项目操作系统”：Obsidian 负责存放资料，Claudian / Claude 负责执行任务，`CLAUDE.md` 负责约束 AI 怎么读、怎么写、什么时候停。

## 1. 这个框架解决什么

长篇小说项目不是一次提示词能解决的事情。

当小说进入几十章、几百章后，作者真正需要的是：

- 角色状态持续一致。
- 主线、支线、伏笔有人维护。
- AI 不把未来大纲当成已发生事实。
- AI 不自动开始下一章。
- 每章写完后有审稿和返修。
- 章节通过后再更新长期设定。
- 临时角色、临时地点、临时物品不会被随手升级成核心设定。

`vaultforge-claude-zh` 把这些要求写进 `CLAUDE.md`、目录结构和岗位 Skill，让 AI 在小说项目里按流程协作。

## 2. 适用环境

推荐环境：

- Obsidian：管理小说 Vault。
- Claudian / Claude：读取 Vault 并执行写作任务。
- Git / GitHub：可选，用于版本管理和备份。

你不需要安装 Obsidian 插件，也不需要运行脚本。这个项目本质上是一套 Markdown 文件模板和 AI 操作协议。

## 3. 快速安装

### 3.1 新建小说 Vault

在 Obsidian 中创建一个新的 Vault，例如：

```text
我的长篇小说/
```

### 3.2 复制框架文件

从本仓库复制：

```text
CLAUDE.md
templates/ 内的所有文件夹和文件
```

复制到你的小说 Vault 根目录。

复制后，Vault 根目录大致如下：

```text
CLAUDE.md
00_System/
01_World/
02_Characters/
03_Plot/
04_Plans/
05_Outputs/
06_Review/
07_Templates/
08_Archive/
09_SKILLS/
10_WritingSupport/
```

### 3.3 第一次只读检查

在 Claudian / Claude 中打开这个 Vault 后，先不要写正文，先让 AI 检查结构：

```text
请读取 CLAUDE.md 并检查当前小说 Vault 结构。不要修改任何文件，只报告缺失文件、设置风险和下一步建议。
```

如果 AI 报告缺失文件，先补齐系统文件，再开始正式写作。

## 4. 初始化小说项目

初始化时，不要一口气填完整个世界观。先填足够支撑前 3 到 10 章的资料。

### 4.1 填写 ProjectConfig

文件：

```text
00_System/ProjectConfig.md
```

建议填写：

- 小说名称。
- 类型题材。
- 目标平台。
- 目标读者。
- 核心卖点。
- 单章字数。
- 预计章节数。
- 禁止事项。

示例：

```markdown
# ProjectConfig

## 基本信息

- 项目名称：示例长篇小说
- 项目类型：都市奇幻长篇小说
- 目标平台：网文平台 / 自媒体 / 私人创作
- 目标读者：喜欢快节奏剧情和强人物关系的读者
- 单章字数：2000-3000 字
- 预计章节数：200 章

## 核心卖点

- 主角从低位起步。
- 隐藏身份逐步揭开。
- 每 3-5 章形成一个小高潮。
```

### 4.2 填写 ProjectState

文件：

```text
00_System/ProjectState.md
```

这是当前状态的单一事实源。

起步阶段可写：

```markdown
| 字段 | 值 |
|---|---|
| 当前阶段 | setup |
| 当前任务单元 | 第001章 |
| 下一允许任务单元 | 第001章 |
| 执行模式 | `single_unit` |
| 是否批量授权 | `false` |
| 是否等待确认 | `false` |
```

注意：不要把未来章节计划写成当前事实。

### 4.3 填写 WritingRules

文件：

```text
00_System/WritingRules.md
```

这里写本书专属规则，例如：

- 单章字数。
- 视角。
- 节奏。
- 爽点类型。
- 禁止水文。
- 角色说话风格。
- 平台限制。

建议先写简版：

```markdown
## 产出要求

- 每章 2000-3000 字。
- 开篇 300 字内进入场景。
- 每章至少有一个冲突、一个推进点、一个章末钩子。

## 禁止事项

- 不写总结式结尾。
- 不用大段设定说明开场。
- 不为了凑字重复心理活动。
```

### 4.4 填写 CharacterRegistry

文件：

```text
00_System/CharacterRegistry.md
```

先登记主角和前几章必出的角色即可。

不要把还没有正式登场的人物全部写满。长篇项目里，过早登记太多角色会增加维护成本。

### 4.5 填写 EntityRegistry

文件：

```text
00_System/EntityRegistry.md
```

登记：

- 核心地点。
- 重要势力。
- 关键物品。
- 能力体系。
- 世界观规则。

原则：只登记已经确认会长期使用的实体。

### 4.6 填写主线和伏笔

文件：

```text
03_Plot/Mainline.md
03_Plot/Foreshadowing.md
```

起步时只需要写：

- 主角长期目标。
- 第一卷核心矛盾。
- 前 10 章必须建立的钩子。
- 已知伏笔和预计回收范围。

## 5. 第一次试跑

建议不要直接让 AI 写 10 章。先跑一个最小闭环。

### 5.1 只规划第 1 章

提示词：

```text
请根据 CLAUDE.md 和当前项目资料，只规划第 1 章。不要写正文。完成后停止。
```

你应该得到：

- 本章目标。
- 核心冲突。
- 场景节点。
- 爽点或情绪点。
- 章末钩子。
- 风险提示。
- 待确认事项。

如果规划里出现你不认可的设定，先改规划，不要进入正文。

### 5.2 写第 1 章正文

确认章纲后，再使用：

```text
请根据 CLAUDE.md，执行 WriteChapter，只写第 1 章正文。不要审稿，不要更新项目记忆，完成后停止。
```

这样可以单独检查写作质量。

### 5.3 审稿

正文完成后：

```text
请根据 CLAUDE.md，执行 ReviewChapter，审查第 1 章正文。不要修改正文，不要更新项目记忆。
```

审稿应给出：

- 是否通过。
- 字数是否合格。
- 章纲是否履约。
- 人物是否 OOC。
- 是否有水文或 AI 腔。
- 是否允许进入项目更新。

### 5.4 返修

如果未通过：

```text
请根据刚才的审稿问题，只返修第 1 章正文。不要新增核心设定，不要更新项目记忆。
```

返修最多建议两轮。两轮仍不通过时，应回到章纲或规则层检查问题。

### 5.5 更新项目记忆

通过审稿后：

```text
请根据 CLAUDE.md，执行 UpdateProject。只把第 1 章已经发生的事实更新到项目记忆，不要写第 2 章。
```

更新内容包括：

- 角色最近状态。
- 新增地点或实体。
- 主线推进。
- 支线变化。
- 伏笔新增或推进。
- 待确认事项。

## 6. 单章完整闭环

当你确认流程稳定后，可以直接使用完整闭环：

```text
请完整执行第 N 章闭环：准备上下文、写正文、审稿、必要时返修、通过后更新项目记忆，然后停止。不得进入第 N+1 章。
```

完整闭环适合日常写作。

它应该按顺序执行：

```text
上下文准备
→ WriteChapter
→ 硬指标检查
→ ReviewChapter
→ 必要返修
→ UpdateProject
→ 汇报并停止
```

最终汇报建议包括：

```text
章节：
正文文件：
实际字数：
审核结论：
返修次数：
项目更新文件：
待确认事项：
状态：
```

## 7. 批量写作

批量写作风险高，建议谨慎使用。

只有在大纲稳定、角色状态清楚、前几章闭环已经跑通后，才授权批量。

推荐提示词：

```text
请按 CLAUDE.md 批量执行第 5 章到第 7 章。逐章串行完成：写作、审稿、必要返修、项目更新。任一章未通过或遇到阻塞，立即停止。不得超过第 7 章。
```

批量任务必须明确：

- 起始章节。
- 结束章节。
- 是否允许连续进入下一章。
- 遇到失败时是否停止。

不要只说：

```text
继续往下写。
```

这会让范围变得模糊。

## 8. 如何管理新增信息

长篇小说最容易膨胀。

每次正文中出现新人物、新地点、新组织、新物品、新能力、新规则，都不要立刻正式化。

先进入：

```text
03_Plot/NewInformationPool.md
```

推荐状态：

| 状态 | 用法 |
|---|---|
| 临时出现 | 路人、一次性地点、短期道具 |
| 观察中 | 可能后续复用，但还不确定 |
| 待升级 | 已经有剧情功能，需要作者确认 |
| 正式设定 | 长期使用，进入 Registry |
| 休眠 | 暂时不用 |
| 废弃 | 不再使用 |

提示词：

```text
请审查第 N 章新增信息。只判断哪些应该临时出现、观察中、待升级或正式设定，不要直接升级需要作者确认的内容。
```

## 9. 如何维护伏笔

伏笔应写入：

```text
03_Plot/Foreshadowing.md
```

每条伏笔建议包含：

- ID。
- 铺垫内容。
- 首次出现章节。
- 最近推进章节。
- 预计回收范围。
- 当前状态。

示例：

```markdown
| ID | 铺垫 | 首次 | 最近 | 预计回收 | 状态 |
|---|---|---|---|---|---|
| F-001 | 主角梦中反复出现黑塔 | 001 | 003 | 010-015 | 活跃 |
```

审计提示词：

```text
请只读审计当前伏笔表，找出超期未推进、重复功能、回收风险和需要作者确认的伏笔。不要修改文件。
```

## 10. 如何做阶段审计

建议每 10 章或 20 章做一次小审计，每 50 章做一次大审计。

提示词：

```text
请根据 CLAUDE.md 执行 AuditProject，对第 1-20 章进行只读阶段审计。重点检查角色断线、伏笔超期、支线过多、设定膨胀、项目状态不同步和章节质量波动。不要修改文件。
```

审计后再分批授权修复：

```text
请只修复审计报告中的 P1 问题：同步 CharacterRegistry 中最近出场章节。不要修改正文和章纲。
```

## 11. 常用提示词

### 只读检查

```text
请读取 CLAUDE.md 并检查当前小说 Vault 结构。不要修改任何文件，只报告缺失文件、设置风险和下一步建议。
```

### 规划单章

```text
请根据 CLAUDE.md，只规划第 N 章。不要写正文。完成后停止。
```

### 写正文

```text
请根据 CLAUDE.md，执行 WriteChapter，只写第 N 章正文。不要审稿，不要更新项目记忆。
```

### 审稿

```text
请根据 CLAUDE.md，执行 ReviewChapter，审查第 N 章正文。不要修改正文，不要更新项目记忆。
```

### 完整单章闭环

```text
请完整执行第 N 章闭环：准备上下文、写正文、审稿、必要时返修、通过后更新项目记忆，然后停止。不得进入第 N+1 章。
```

### 项目更新

```text
请根据已通过审稿的第 N 章执行 UpdateProject。只记录已经发生的事实，不要登记未来计划，不要写下一章。
```

### 阶段审计

```text
请执行只读 AuditProject，审计第 X 章到第 Y 章。重点检查角色、实体、伏笔、支线、项目状态和质量波动。不要修改文件。
```

## 12. 常见问题

### AI 想自动写下一章怎么办？

检查提示词里是否写了“继续”“往下写”“推进后续”。建议明确加：

```text
完成当前章后停止，不得进入下一章。
```

### AI 把未来大纲当成事实怎么办？

把这条问题记录到：

```text
03_Plot/Contradictions.md
```

并在提示词中强调：

```text
章纲和未来计划不能登记为已发生事实。
```

### 角色越来越多怎么办？

先审查 `NewInformationPool.md`。把一次性角色标记为“临时出现”或“废弃”，不要都升级进 `CharacterRegistry.md`。

### 章节变水怎么办？

检查 `WritingRules.md` 是否有硬指标，例如：

- 每章必须有冲突。
- 每章必须有推进。
- 每章必须有章末钩子。
- 禁止重复心理活动凑字。

然后用 `ReviewChapter` 找出具体水文位置。

### Claudian 每次读太多文件怎么办？

检查 `CLAUDE.md` 和 Skill 中的上下文读取规则。提示词中可以加：

```text
请按最小上下文切片读取，不要全文读取无关角色卡、历史正文和全部审稿记录。
```

## 13. 推荐使用节奏

新项目：

```text
初始化资料
→ 规划前 3 章
→ 单独写第 1 章
→ 单独审第 1 章
→ 更新项目记忆
→ 再尝试完整闭环
```

稳定项目：

```text
每天 1 章完整闭环
每 5 章检查一次新增信息池
每 10-20 章做一次小审计
每 50 章做一次大审计
```

## 14. 最重要的使用原则

```text
不要让 AI 一次做太多事。
不要让 AI 默认读取全库。
不要让 AI 把计划当事实。
不要让 AI 自动进入下一章。
不要跳过审稿就更新项目记忆。
```

这个框架的价值不在于让 AI 更自由，而在于让 AI 在长篇小说项目里更可靠。
