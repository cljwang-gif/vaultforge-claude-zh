# 编写 Skill 文件

Skill 文件位于 `09_SKILLS/`。

每个 Skill 应该只定义一个岗位、一个阶段。

## 推荐结构

```markdown
---
skill: SkillName
role: 清晰的岗位名称
updated: YYYY-MM-DD
---

# SkillName

## 职责

## 权限

## 上下文

## 执行流程

## 质量门槛

## 输出格式

## 停止条件
```

## 好的岗位边界

一个好的 Skill 只做一件事：

- `CreateOutline`：负责规划，不写正文。
- `WriteChapter`：负责生成产出，不更新长期记忆。
- `ReviewChapter`：负责审稿，不擅自改写。
- `UpdateProject`：负责更新项目记忆，不写正文。
- `AuditProject`：负责全局体检，未授权不修复。

## 权限写法

建议明确写“应该”和“不得”。

示例：

```markdown
## 权限

应该：

- 读取当前规划。
- 生成当前授权产出。
- 捕获新增信息，交给后续更新。

不得：

- 修改项目规则。
- 更新 Registry。
- 开始下一个产出。
```

## 上下文规则

每个 Skill 都应该说明它可以读取什么。

避免写“读取所有项目文件”。优先写最小上下文切片。

## 停止条件

每个 Skill 都应该定义何时停止：

- 缺少上下文。
- 事实冲突。
- 质量门槛失败。
- 需要用户确认。
- 授权范围完成。
